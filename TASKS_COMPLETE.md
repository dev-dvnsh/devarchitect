## Task 9 — Workflow Enforcement

### Purpose

Right now nothing stops a developer from running `devarchitect roadmap`
before running `devarchitect init`. That would crash or produce garbage data.
Workflow enforcement makes devarchitect behave like a real structured tool —
commands must run in the correct order, just like you cannot push to Git
before committing. This is what separates devarchitect from a simple
note-taking CLI.

### Prerequisite Check Pattern

Each command checks the ones before it:

- analyze → needs vision.json
- stack → needs vision.json + analysis.json
- roadmap → needs vision.json + analysis.json + techstack.json
- decision → needs vision.json
- progress → needs vision.json + roadmap.json
- export → needs vision.json (others are optional, skip gracefully)
- status → needs nothing, always runs

### Learn First — search these, 20 mins max

1. javascript extract reusable function
   You are going to write one reusable checkPrereqs() function
   that all commands can call instead of repeating the same
   if/existsSync block in every file.
   Search: javascript reusable utility function module

2. nodejs module exports multiple functions
   You will put checkPrereqs() in a shared utility file called
   src/utils.js and export it so all commands can import it.
   Search: nodejs module exports multiple functions example

### Jot Down in Notebook

- What a utility file is — a shared file that holds functions used
  by many other files. Keeps code DRY (Don't Repeat Yourself).
- The DRY principle — one of the most important principles in
  software engineering. Never write the same logic twice.
  Put it in one place and reuse it everywhere.

### What to Build

1. Create src/utils.js
2. Write a checkPrereq(filename, commandName) function that:
   - Takes a filename like "vision.json" and a command name like "init"
   - Checks if .devarchitect/filename exists
   - If not, prints: "Please run devarchitect [commandName] first"
   - Then calls process.exit(1)
3. Export it from utils.js
4. Replace all existing prereq checks in init.js, analyze.js,
   stack.js, roadmap.js, decision.js, progress.js with calls
   to this function

### You Will Know It Works When

You delete .devarchitect/ and run:
devarchitect roadmap
It should tell you to run init first, not crash with an ugly error.

init done
analyse done
stack done
roadmap done

## Task 11 — Versioning .bak Files

### Purpose

Right now if you run devarchitect stack twice, the second run silently
overwrites the first. The developer loses their previous state with no
way to recover it. This task adds a simple versioning system — before
overwriting any JSON file, save the old version as a .bak file.
This is devarchitect's equivalent of Git's ability to go back in time.

### Learn First — search these, 20 mins max

1. nodejs fs.copyFileSync example
   You will copy the existing file to a .bak version before overwriting.
   Search: nodejs fs copyFileSync example

2. javascript Date timestamp filename safe
   You will add a timestamp to the .bak filename so multiple backups
   don't overwrite each other.
   Search: javascript date toISOString replace colon filename safe

### Jot Down in Notebook

- What fs.copyFileSync does — copies a file from one path to another
- Why versioning matters — any production system that stores state
  needs a way to recover from mistakes. This is the simplest possible
  versioning system.
- The .bak filename pattern:
  vision.json → v.json.2026-04-06T09-12-39.bak

### What to Build

1. Add a backupIfExists(filePath) function to src/utils.js
2. It should:
   - Check if the file already exists
   - If yes copy it to [filename].[timestamp].bak in the same folder
   - If no do nothing
3. Call backupIfExists() in every command before writing its JSON file

### You Will Know It Works When

You run devarchitect stack twice with different answers.
Inside .devarchitect/ you should see:

- techstack.json (latest)
- techstack.json.2026-04-06T09-12-39.bak (previous version)

## Task 13 — Setup Express Server

### Purpose

The CLI stores data as JSON files. The React dashboard needs to read
that data over HTTP. This Express server is the bridge between them —
it reads the .devarchitect/ JSON files and exposes them as REST API
endpoints. It is a tiny server with one job: serve the project data
to whoever asks for it.

### Learn First — search these, 20 mins max

1. express js basic server setup nodejs
   You already know Express from MERN. This is the same thing.
   Search: express js hello world example

2. express js serve static files
   The server will also serve the React build files so the browser
   can load the dashboard.
   Search: express static files middleware

3. cors npm nodejs
   The React app runs on a different port than the server during
   development. CORS allows them to talk to each other.
   Search: cors npm express example

### Jot Down in Notebook

- What CORS is — Cross Origin Resource Sharing. Browsers block
  requests between different origins (ports/domains) by default
  for security. CORS headers tell the browser to allow it.
- The difference between development and production server setup:
  Dev: React on port 5173, Express on port 3001, CORS needed
  Prod: Express serves the React build, same port, no CORS needed

### What to Build

1. Create src/server/index.js
2. Install express and cors:
   npm install express cors
3. Build these endpoints:
   GET /api/status — returns which JSON files exist
   GET /api/vision — returns vision.json contents
   GET /api/analysis — returns analysis.json contents
   GET /api/stack — returns techstack.json contents
   GET /api/roadmap — returns roadmap.json contents
   GET /api/decisions — returns decisions.json contents
   GET /api/progress — returns progress.json contents
4. Each endpoint should return { data: null, message: "not found" }
   if the file does not exist — never crash on missing files

### You Will Know It Works When

You run:
node src/server/index.js
Then open browser at <http://localhost:3001/api/vision>
You see your vision.json data returned as JSON in the browser.
